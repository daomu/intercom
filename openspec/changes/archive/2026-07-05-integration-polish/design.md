## Context

前 16 个变更已交付完整可运行固件：BSP 驱动（02）、Storage/Crypto/Network/Audio/Display/Power Service（03-06）、App Shell/Settings（07）、对讲包格式（08）、三阶段组队（09）、PTT 通话（10）、jitter/混音（11）、恢复/心跳（12）、App UI（13）、变声器（14）、熄屏/电源（15）、安全诊断（16）。当前需要在本期完成端到端贯通、长稳、调优与验收，作为 17-change 路线图终点。

PRD §26 列出 8 类验收标准（§26.1-26.8），§29 列出 9 条成功标准，技术设计 §21 列出 10 项"风险与待验证项"全部标注"待验证"为本期任务。PRD §20 音频焦点优先级与规则在前期实现中已落地，本期需在真实多音频源场景下验收。PRD §27 开发优先级 P0-P3 全部能力均已实现，本期做最终核验。

## Goals / Non-Goals

**Goals:**
- 2 设备端到端全流程（boot → 组队 → PTT → 熄屏收/发 → reboot → 冷启动 restore）一次走通无回归
- PRD §26.1-26.8 全量验收清单逐项核验并记录结果
- PRD §26.8 五项长稳场景全部通过（长时间待机通话 / 反复熄屏唤醒 / 连续重启恢复 / 异常数据不卡死 / 多设备长时间在线成员状态稳定）
- 技术设计 §21 待验证项表全部实机验证并记录结论（含 Opus PLC 4 帧阈值、jitter 3 帧初始水位、Opus 比特率三项调优结果）
- PRD §20.1/§20.2 音频焦点优先级与三条规则在真实场景下验收通过

**Non-Goals:**
- 新增任何业务功能或 spec 级行为（本期为验证/调优）

- 4 设备压力测试的稳定性保证（best-effort，记录现象不作为交付阻塞项）
- 修复前期变更的设计缺陷（如发现，单独立项）

## Decisions

### D1：验收策略 = 场景化人工核验 + 日志归档
PRD §26 验收清单为行为级断言，适合人工按场景操作设备并对照预期。每个场景记录：操作步骤、设备 A/B 串口日志关键行、UI 截图/照片、判定结论（PASS/FAIL/已知可接受退化）。不引入自动化测试框架（ESP32 on-device 自动化投入产出比低，且 PRD §23 已明确"近同时竞争短时退化"等场景属可接受退化）。备选：CI 自动化 HIL——硬件在环自动化成本高且不稳定，排除。

### D2：调优范围 = 三参数，稳定优先
本期调优仅限技术设计 §21 明确标注"实测调优"的三项：
1. Opus PLC 静音封底阈值（PRD §16.5 默认 4 帧/80ms）：在可控丢包率（5%/10%/20%）下主观评估连续性，必要时微调 ±1 帧
2. jitter 初始水位（PRD §16.6 默认 3 帧/60ms）：弱网下评估首字丢失与端到端延时，必要时微调 ±1 帧
3. Opus 比特率（技术设计 §10.4 "实测调优，优先稳定"）：在 RISC-V 160MHz 无 PSRAM 下测编码耗时与音质，选定稳定值
调优结果若偏离 BoardProfile 默认值，以最小常量 patch 回写。备选：保持默认不调——但 §21 明确要求"实测调优"，不调无法闭环。排除。

### D3：长稳测试 = 分场景独立通过判定
PRD §26.8 五项长稳场景各自独立判定 PASS/FAIL，不要求同时满足。每项定义通过判据：
- 长时间待机通话：待机 ≥2h 后 PTT 通话成功
- 反复熄屏唤醒：≥20 次熄屏/唤醒循环后对讲正常
- 连续重启恢复：≥3 次重启后组关系正确恢复
- 异常数据不卡死：写入损坏 NVS 后启动不卡死、回退默认/清理
- 多设备长时间在线：≥2h 在线，成员在线状态无错误抖动
备选：缩短时长——但 PRD §26.8 "长时间"语义需可观测，2h 为可执行最低门槛。

### D4：3/4 设备压力测试 = best-effort（2 设备 + 代码分析），不阻塞交付
3 设备混音与 4 设备压力测试（6 对 LMK、3 路混音、peer 表 12 表项）按 PRD §28 风险表属"需验证"但自由模式 3 路混音在 PRD §13.2/§26.3 已标注"best-effort"。当前仅 2 台设备可用，以 2 设备实机 + 代码分析覆盖（审查混音路径、ECDH 全网状、peer 表容量）；完整 3/4 设备实机验证推迟至未来硬件可用时。结论作为已知边界记录，不作为一期交付阻塞项。备选：要求 4 设备全稳定——超出 PRD 一期范围，排除。

### D5：音频焦点验收 = 真实多音频源场景
PRD §20 优先级表 5 级 + 3 条规则需在真实多音频源场景下验收。由于 ESP32-C6 一期无第三方 App 音频播放能力（PRD §24 仅暴露对讲/Settings），"其他 App 音频被压制"规则以模拟方式验证：在播放对讲语音期间触发提示音/变声预览，验证优先级压制与互斥规则。备选：纯代码审查——无法验证运行时行为，排除。

### D6：调优 patch 回写原则 = 仅常量，不动逻辑
若调优确证需变更阈值，仅修改 `BoardProfile` 中的 `const`（如 `PLC_CONSECUTIVE_LOSS_THRESHOLD`、`JITTER_INIT_FRAMES`、`OPUS_BITRATE`），Service 实现引用常量自动生效。不修改算法逻辑、不改变 spec 级行为契约。备选：硬编码调优值——违反技术设计 §2 常量集中原则，排除。

## Risks / Trade-offs

- **[端到端场景发现前期变更回归缺陷]** → 若发现，记录缺陷并评估是否阻塞交付；严重缺陷单独立项修复后重新跑本变更场景
- **[调优参数无客观基准仅主观听感]** → 由 ≥2 人主观评估一致通过；调优前后 A/B 对比；记录样本日志
- **[4 设备压力测试资源不足]** → 2 设备实机 + 代码分析覆盖；完整 3/4 设备实机验证推迟至未来硬件可用时，在验收记录中标注
- **[长稳测试耗时占交付周期长]** → 2h 待机 + 多次重启等总耗时约 1-2 个工作日；并行执行可压缩
- **[弱网环境不可控复现]** → 用 `tc`/ iptables 在 Linux AP 侧模拟丢包，或物理距离/遮挡方式制造弱网；记录场景参数
- **[音频焦点"其他 App 音频"在一期无真实 App]** → D5 模拟验证，验收记录标注"一期无第三方 App，规则以模拟方式验证"
- **[调优导致 BoardProfile 常量变更影响前期变更 spec]** → 仅常量值变更不改 spec 行为契约；若需改 spec 级阈值（如 PLC 从 4 帧改 5 帧），视为 spec 修订需另开变更

## Migration Plan

本期为验证/调优，无运行时迁移。部署步骤：
1. 拉取前 16 个变更全部代码，构建固件
2. 2 台设备烧录最新固件
3. 按 tasks.md 场景顺序执行验收
4. 调优参数确证后回写 BoardProfile 常量 patch（若有）
5. 验收记录归档至 `docs/acceptance/integration-polish-acceptance.md`
6. 全部 PASS 后一期正式需求基线交付

回滚：调优 patch 若引入回归，`git revert` 回到默认常量值；验收记录保留。
